# Agent Skill 基础设施化——整合方案 v3

## Executive Summary

本报告整合了三份前置调研（可行性分析、消费者侧解决方案、开发测试实践调研）的全部发现，提出一套覆盖 Skill 全生命周期的基础设施方案。方案围绕两条产品线展开：**安全审计 Skill**（消费者侧：Skill 用户安装前的风险感知 + 安装后的变更感知）和**开发工具 Skill**（生产者侧：Skill 作者的开发脚手架 + 行为测试运行器）。两条产品线共享分层架构（L1 纯文档 / L2 脚本化 / L3 LLM 增强），每层独立可用、渐进增强。基于现有生态成熟度评估（格式校验 6/10，行为测试 2/10，安全审计 1/10），最有价值的切入点是**行为测试运行器**——当前唯一的完全空白点。

---

## 一、从三份报告到整合方案

### 1.1 三份报告的核心发现回顾

**可行性报告**（feasibility）识别了四个核心能力缺口和对应的严重度：

| 能力缺口 | 严重度 | 现状 |
|---------|--------|------|
| 版本管理缺失 | 8/10 | 安装后无法锁定版本，更新可能引入行为变更 |
| 测试框架空白 | 7/10 | Skill 是自然语言+代码混合体，无标准测试方法 |
| 安全审计缺位 | 9/10 | ClawHavoc 事件（1184个恶意 Skill），36.82%漏洞率 |
| 跨平台不兼容 | 7/10 | 各平台目录结构、加载机制、权限模型各不相同 |

可行性报告还规划了三阶段产品路径：MVP（安全审计 CLI）→ 第二阶段（版本管理 + 测试框架 + Registry）→ 第三阶段（企业治理面板）。本方案沿用 MVP 优先的策略，但将"测试框架"从第二阶段提前到核心规划中。

**消费者侧解决方案报告**（solutions v2）聚焦"安全审计 + 版本感知"，通过三轮讨论收敛为：

- 个人开发者定位（砍掉企业级）
- 预警雷达定位（安装前风险感知，不包办深度审计）
- 零侵入适配（依赖声明可选，推断+容错兜底）
- 分层 Agent Skill 产品形态（L1/L2/L3），非独立 CLI

**开发测试实践调研报告**（dev-test practices）发现生态成熟度仅 3/10：

- 开发工具 2/10（无脚手架、无模板、无实时反馈）
- 格式校验 6/10（agent-skill-linter 17条规则，但只查格式不查行为）
- 行为测试 2/10（skill-creator eval 是唯一自动化工具，覆盖窄、无 CI）
- 质量评估 3/10（helloandy 8维度、qa-agent-testing 6维度，但都非自动化工具）

### 1.2 原方案的覆盖范围缺口

消费者侧解决方案（solutions v2）的四层架构覆盖了：

1. 标准化规范层 → 部分覆盖（依赖声明格式设计，但未覆盖触发优化、输出规范等）
2. 依赖感知层 → 已覆盖（声明→推断→容错三层渐进）
3. 安全预警层 → 已覆盖（YARA快扫→LLM意图分析→人话摘要）
4. 版本感知层 → 已覆盖（变更摘要→行为差异→依赖校验→回滚→更新提醒）

但 feasibility 报告中的"测试框架空白"和"开发工具链缺失"在这套架构中没有独立的位置。版本感知层的 Snapshot Testing 是版本管理的子能力（检测更新后行为是否退化），不是 Skill 作者在开发阶段使用的"行为测试运行器"。

### 1.3 整合后的双线产品架构

基于以上分析，整合方案采用**双线产品架构**——两条独立的 Skill 产品线，共享底层设计理念（分层、渐进、零侵入），但服务不同的用户群体和不同的使用场景：

| 维度 | 产品线 A：安全审计 Skill | 产品线 B：开发工具 Skill |
|------|------------------------|------------------------|
| 目标用户 | Skill 消费者（安装和使用 Skill 的人） | Skill 生产者（编写和维护 Skill 的人） |
| 核心价值 | 安装前的风险感知 + 安装后的变更感知 | 开发阶段的快速反馈 + 行为正确性验证 |
| 对应痛点 | 安全审计缺位（9/10）+ 版本不可控（8/10） | 测试困难（7/10）+ 开发工具链缺失（开发工具2/10） |
| 核心能力 | 安全扫描 + 变更摘要 + 依赖校验 | 脚手架 + 格式校验 + 行为测试 |
| 竞争空白 | 完全空白（朱雀扫描器和 x-security 都是纯文档） | 行为测试完全空白（格式校验已有 agent-skill-linter） |

## 二、共享的分层架构原则

两条产品线遵循相同的分层设计原则：

### 2.1 L1 纯文档层：零依赖，即装即用

纯 SKILL.md 格式，所有能力通过指导 Agent 使用其内置工具（文件读取、文本分析、bash 执行）来实现。不调用任何外部工具、不需要任何额外依赖。可以在所有 Agent 平台上使用，可以通过 x-cmd/skill 仓库分发。

L1 的核心价值是**零门槛体验**——用户/作者装了就能用，不需要额外配置。L1 也是回馈社区的层——基础能力可以注册进 x-cmd/skill 仓库，让所有 x-cmd 用户受益。

### 2.2 L2 脚本化层：可控依赖，能力增强

SKILL.md + scripts/ 目录（Python）。通过 Python 脚本提供 L1 无法实现的能力（YARA 规则匹配、Agent 运行时调用、自动化测试执行等）。依赖 Python 3 环境（大多数开发者已有），`scripts/setup.sh` 自动检测并安装 uv 包管理器和依赖包。

L2 的核心价值是**能力跃升**——从"指导 Agent 手工操作"到"自动化执行"。分发通过 GitHub Release 和 Agent 平台手动安装。

### 2.3 L3 增强层：可选高级能力，最强差异化

在 L2 基础上接入 LLM API（Claude/GPT），提供语义理解、意图分析、质量评分等高级能力。需要用户配置 API Key。

L3 的核心价值是**智能增强**——利用 LLM 处理自然语言的模糊性和语义等价性，这是纯规则/脚本无法做到的。

### 2.4 层级间的关系

每一层独立可用。L1 装了就能用，不需要等 L2；L2 装了就能深度使用，不需要等 L3。用户/作者根据自身环境和需求选择合适的层级。

层级间是**增强关系**而非**依赖关系**。L3 不会因为 L1 不存在就无法工作——每层都是完整的功能闭环。

## 三、产品线 A：安全审计 Skill（消费者侧）

### 3.1 定位

面向 Skill 消费者（安装和使用 Skill 的开发者），提供"安装前的风险雷达"和"安装后的变更感知"。不替代专业安全审计，不做运行时权限拦截，不做对抗性红队测试——只是帮用户在安装 Skill 前快速了解风险，在更新 Skill 前了解变更。

### 3.2 分层能力设计

**L1 基础扫描（纯文档）：**

Agent 读取目标 Skill 的 SKILL.md 文本，按结构化步骤分析：
- 检查 frontmatter 完整性（name、description、author）
- 检查是否有权限声明（`permissions` 字段）和敏感操作关键词（`rm -rf`、`curl | sh`、`sudo`、`eval`、`exec`）
- 检查是否有外部网络请求（URL、API endpoint、webhook）
- 检查 description 是否包含触发关键词（评估触发准确性）
- 输出：结构化风险摘要（高/中/低风险，附带发现的具体风险点）

类似朱雀实验室 AI-Infra-Guard 和 x-cmd x-security 的纯文档扫描方式。零依赖，任何 Agent 加载后即可使用。

**L2 深度扫描（脚本化）：**

在 L1 基础上，Python 脚本执行：
- YARA 规则匹配：对 SKILL.md 和 scripts/ 目录中的所有文件进行模式匹配（恶意模式、混淆代码、数据外泄模式）
- 文件权限检查：扫描 scripts/ 中是否有设置可执行权限的脚本、是否修改系统文件
- 依赖声明静态分析：解析 SKILL.md 中的 `dependencies` 字段，检查是否有已知漏洞的依赖包
- references/ 路径可达性检查：验证 references/ 中引用的文件是否存在

**L3 意图分析（LLM 增强）：**

在 L2 基础上，调用 LLM API 进行深层意图推理：
- Cisco Scanner 第5层（Claude 3.5 Sonnet）：对 SKILL.md 进行意图分析，识别隐藏的恶意意图
- GovTech 五步结构化推理：提升安全判断准确性（F1=0.844）
- 自然语言风险摘要：将技术发现翻译为开发者可理解的风险说明

### 3.3 版本感知能力

版本感知能力跨层级实现：

| 能力 | L1 | L2 | L3 |
|------|----|----|-----|
| 检测可用更新 | Git fetch + SHA 对比 | 同 L1 | 同 L1 |
| 变更摘要 | Git diff 读取（Agent 展示） | + 自动 diff 解析 | + LLM 生成自然语言变更摘要 |
| 依赖差异检测 | 对比新旧 dependencies 字段 | + pip/npm 依赖树对比 | 同 L2 |
| 一键回滚 | Git checkout（Agent 执行） | 同 L1 | 同 L1 |
| 行为差异检测 | 无 | Snapshot 对比（文本 diff） | + LLM 语义等价判断 |

### 3.4 分发策略

- L1：注册进 x-cmd/skill 仓库索引，`x skill add audit` 一条命令安装
- L2：GitHub Release + Agent 平台手动安装
- L3：同 L2，额外需要 API Key 配置

## 四、产品线 B：开发工具 Skill（生产者侧）

### 4.1 定位

面向 Skill 生产者（编写和维护 Skill 的开发者），提供"开发阶段的快速反馈"和"行为正确性验证"。这是 feasibility 报告中"测试框架空白"和 dev-test practices 报告中"行为测试完全空白"的直接回应。

### 4.2 分层能力设计

**L1 开发引导（纯文档）：**

Agent 读取作者的 SKILL.md，按结构化步骤进行开发阶段的质量检查：
- **触发优化检查**：评估 description 是否包含足够具体的触发关键词，是否可能导致误触发或漏触发（参考 skill-creator 的描述优化方法论）
- **输出规范检查**：检查 SKILL.md 是否明确定义了输出格式、示例、约束条件（解决"输出不一致"痛点）
- **容错覆盖检查**：扫描 SKILL.md 中的多步流程，检查是否有容错处理（解决"一出错就崩溃"痛点）
- **Token 效率检查**：评估 SKILL.md 是否遵循渐进式披露（<500行），是否有过长内容应拆分到 references/
- **helloandy 8维度快速评分**：结构化评估 Skill 质量（触发精度、输出完整性、铁律具体性、错误恢复覆盖、示例质量、简洁性、一致性、边缘情况处理）

输出：结构化质量报告，标注改进建议。

**L2 脚手架 + 格式校验 + 行为测试（脚本化）：**

L2 是开发工具 Skill 的核心层，提供三组自动化能力：

（a）**脚手架能力**：
- `skill init <name>`：基于模板生成 Skill 目录结构（SKILL.md 模板、references/ 目录、scripts/ 目录、evals/ 目录、.gitignore、LICENSE）
- `skill dev`：监听 SKILL.md 变更，自动同步到 Agent 技能目录（热重载开发体验）

（b）**格式校验集成**：
- 直接调用 agent-skill-linter（17条规则，不做重复造轮子）
- 增加 content-aware 检查（description 质量评估、输出规范完整性检查）

（c）**行为测试运行器**（最有价值的切入点）：

这是当前生态的完全空白点——没有现有工具能自动验证"Skill 被触发后是否按预期执行"。

核心能力设计：

| 步骤 | 能力 | 实现方式 |
|------|------|---------|
| 1. 测试用例定义 | 支持 evals.json 格式（兼容 skill-creator） | JSON Schema 校验 |
| 2. 自动运行 | 给定 Skill 路径 + 测试套件，启动 Agent 运行每个用例 | 调用 Claude Code / OpenCode 的 CLI 接口 |
| 3. 触发判定 | 正例是否触发、负例是否跳过 | 解析 Agent 日志/轨迹 |
| 4. 流程判定 | Agent 是否按 SKILL.md 定义的步骤执行 | 工具调用序列校验 |
| 5. 输出判定 | 结果是否符合输出规范 | 格式检查 + 内容关键词检查 |
| 6. 快照对比 | 首次保存快照，后续与快照对比 | 文本 diff + 语义等价判断（L3） |
| 7. CI 集成 | JSON 报告 + 退出码 | GitHub Actions / GitLab CI |

测试用例格式（兼容 skill-creator evals.json）：
```json
{
  "skill_name": "my-skill",
  "evals": [
    {
      "id": "should-trigger-positive",
      "type": "positive",
      "prompt": "帮我分析这段代码的性能",
      "expected": {
        "triggered": true,
        "output_contains": ["时间复杂度", "空间复杂度"],
        "tools_used": ["read_file", "search_content"]
      }
    },
    {
      "id": "should-not-trigger-negative",
      "type": "negative",
      "prompt": "今天天气怎么样",
      "expected": {
        "triggered": false
      }
    }
  ]
}
```

**L3 智能增强（LLM 增强层）：**

在 L2 基础上接入 LLM API：

- **语义等价判断**：Snapshot Testing 的核心难题——LLM 输出具有非确定性，逐字对比不可靠。L3 调用 LLM 判断两次输出在语义上是否等价（参考 Agent-as-Judge 方法论）
- **自动生成测试用例**：给定 SKILL.md，LLM 自动生成正例/负例测试 prompt，扩展测试覆盖面（解决 skill-creator "2-3个测试用例"的覆盖不足问题）
- **变形测试**：保留语义的小改动输入（同义词替换、语序调整），验证输出不变性（参考 qa-agent-testing 方法论）
- **质量评分自动化**：将 helloandy 8维度评分和 qa-agent-testing 6维度评分从手工评分升级为 LLM 自动评分

### 4.3 分发策略

- L1：注册进 x-cmd/skill 仓库（开发者 Skill 基础质量检查，任何 x-cmd 用户可用）
- L2：GitHub Release + Agent 平台手动安装（行为测试运行器需要 Agent 运行时支持，分发门槛略高）
- L3：同 L2，额外需要 API Key

## 五、两条产品线的关系

### 5.1 独立但互补

两条产品线在功能上完全独立——安全审计 Skill 不需要开发工具 Skill 就能工作，反之亦然。用户可以只安装其中一条。

但它们在架构上互补：
- **开发工具 Skill** 帮助作者产出更高质量的 Skill → 减少恶意/低质 Skill 的数量 → 间接减轻安全审计 Skill 的负担
- **安全审计 Skill** 的扫描结果（风险模式、常见问题）可以反馈给开发工具 Skill 的质量检查规则 → 形成"开发-审计-改进"的正向循环

### 5.2 共享的工程基础设施

两条产品线共享：
- Git 作为版本控制底座
- x-cmd 作为可选的工程底座和分发通道
- 相同的 L1/L2/L3 分层设计理念
- 相同的 evals.json 测试用例格式（安全审计的"安全测试"和开发工具的"行为测试"可以用相同的格式定义测试用例）

### 5.3 与 x-cmd 的关系

两条产品线都遵循"互补共生"策略：

| 产品线 | L1 与 x-cmd 的关系 | L2/L3 与 x-cmd 的关系 |
|--------|-------------------|---------------------|
| 安全审计 | L1 注册进 x-cmd/skill 仓库，回馈基础安全扫描能力 | 独立分发，YARA+LLM 能力 x-cmd 没有 |
| 开发工具 | L1 注册进 x-cmd/skill 仓库，回馈基础质量检查能力 | 独立分发，行为测试运行器 x-cmd 没有 |

L1 是给 x-cmd 社区的"礼物"——基础能力免费贡献，建立生态位。L2/L3 是差异化能力——x-cmd 没有的东西，保持独立价值和收入潜力。

## 六、与现有生态的整合策略

### 6.1 不重复造轮子的部分

以下能力已有成熟的开源工具，直接整合而不自建：

| 能力 | 现有工具 | 整合方式 |
|------|---------|---------|
| 格式校验 | agent-skill-linter（17规则） | L2 直接调用 CLI |
| 官方规范验证 | agentskills.io skills-ref | L1 指导 Agent 调用 |
| 快速扫描 | AI-Infra-Guard | L2 整合为后端引擎 |
| LLM 意图分析 | Cisco Scanner（L5） | L3 调用 API 或本地部署 |
| 结构化推理 | GovTech 五步推理 prompt | L3 整合为 prompt template |

### 6.2 我们需要自建的部分

以下能力是现有工具的完全空白，必须自建：

| 能力 | 空白程度 | 原因 |
|------|---------|------|
| 行为测试运行器 | 完全空白 | 没有工具能自动验证 Skill 触发后的执行行为 |
| 安全变更摘要 | 完全空白 | 版本更新时的 LLM 生成变更摘要不存在 |
| 测试用例生成 | 完全空白 | LLM 自动生成正/负例测试 prompt 不存在 |
| 开发脚手架 | 接近空白 | Anthropic 有 template/ 但不是 CLI 工具 |
| 触发准确性测试 | 接近空白 | skill-creator 有描述优化但不测试触发准确性 |

### 6.3 需要自建但可借鉴的部分

| 能力 | 借鉴来源 | 差异化 |
|------|---------|--------|
| 质量评分框架 | helloandy 8维度 + qa-agent-testing 6维度 | 从手工评分升级为自动化评分 |
| Snapshot Testing | DeepEval + syrupy | 适配 Skill 场景（语义等价而非逐字对比） |
| 测试用例格式 | skill-creator evals.json | 扩展支持负例、流程校验、安全检查 |

## 七、实施路径

### 7.1 优先级排序

基于生态成熟度评估和空白程度，建议的实施优先级：

| 优先级 | 切入点 | 原因 |
|--------|--------|------|
| P0（最高） | L2 行为测试运行器 | 完全空白、最有差异化价值、开发者痛感最强 |
| P1（高） | L1 开发引导 + L1 安全扫描 | 零依赖、可以快速上线、建立社区存在感 |
| P2（中） | L2 安全深度扫描 | YARA+静态分析，技术成熟度较高 |
| P3（中低） | L2 脚手架 + 热重载 | 有价值但非核心差异化 |
| P4（低） | L3 LLM 增强 | 依赖 API Key，用户门槛高，但差异化最强 |

### 7.2 三阶段实施路径

**第一阶段（0-3个月）：L1 双线 + L2 行为测试运行器原型**

安全审计侧：
- 实现 L1 安全扫描 SKILL.md（结构化风险分析，参考 x-security 和 AI-Infra-Guard）
- 注册进 x-cmd/skill 仓库

开发工具侧：
- 实现 L1 开发引导 SKILL.md（helloandy 8维度 + 触发优化 + 输出规范检查）
- 注册进 x-cmd/skill 仓库
- 实现 L2 行为测试运行器核心：evals.json 解析 + Agent 调用 + 触发判定 + 输出判定
- 输出 JSON 报告 + 退出码，可集成 CI

产出：两个 L1 Skill（x-cmd 可分发）+ 一个 L2 行为测试运行器原型（GitHub Release）

**第二阶段（3-6个月）：L2 安全深度扫描 + L2 脚手架 + 测试运行器完善**

安全审计侧：
- 实现 L2 YARA 规则扫描器
- 实现文件权限检查 + 依赖静态分析
- 接入 AI-Infra-Guard 作为快扫后端

开发工具侧：
- 实现 `skill init` 脚手架 + `skill dev` 热重载
- 行为测试运行器增加流程判定、快照对比、CI 集成完善
- 集成 agent-skill-linter 作为格式校验步骤

产出：完整的 L2 安全审计 Skill + L2 开发工具 Skill（含行为测试运行器）

**第三阶段（6-12个月）：L3 智能增强 + 生态扩展**

安全审计侧：
- 接入 LLM API 进行意图分析（Cisco Scanner L5 + GovTech 推理）
- 实现版本变更摘要的 LLM 生成

开发工具侧：
- 实现语义等价判断（Agent-as-Judge）
- 实现自动测试用例生成
- 实现变形测试
- helloandy/qa-agent-testing 评分的自动化实现

生态侧：
- 向 x-cmd 社区提交 L1 的 PR
- 推动 SKILL.md 规范扩展（可选的 evals/ 目录、dependencies 字段）
- 建立 YARA 规则和测试用例的社区贡献机制

产出：完整的 L1/L2/L3 三层双线 Skill 体系

### 7.3 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 行为测试运行器依赖 Agent 运行时 | Claude Code/OpenCode CLI 接口可能变化或受限 | 先做适配层抽象，不硬编码特定 Agent 的接口 |
| x-cmd 吸收 L1 能力 | GPL-3.0 下社区可能内置类似功能 | L2/L3 保持独立分发，L1 的价值是"建立生态位"而非"收入来源" |
| LLM API 成本 | L3 的 LLM 调用有使用成本 | L3 完全可选，L1/L2 不依赖 LLM |
| 测试用例生态冷启动 | 没人写 evals.json | 从自测开始，为核心 Skill（自己的安全审计 Skill）写测试用例作为示范 |

## 八、决策回顾

### 8.1 本方案如何回应 feasibility 报告的四个缺口

| 缺口 | 对应能力 | 产品线 | 实施阶段 |
|------|---------|--------|---------|
| 版本管理缺失（8/10） | 版本感知（变更摘要+回滚+依赖校验） | A:安全审计 | 第一阶段 L1 + 第二阶段 L2 |
| 测试框架空白（7/10） | 行为测试运行器（触发判定+流程判定+输出判定+快照） | B:开发工具 | 第一阶段 L2 原型 + 第二阶段完善 |
| 安全审计缺位（9/10） | 安全扫描（YARA+意图分析+风险摘要） | A:安全审计 | 第一阶段 L1 + 第二阶段 L2 + 第三阶段 L3 |
| 跨平台不兼容（7/10） | 格式校验（agent-skill-linter 集成）+ 触发优化 | B:开发工具 | 第一阶段 L1 + 第二阶段 L2 |

### 8.2 本方案如何回应三个关键收敛决策

**决策一：个人开发者定位。** 两条产品线都面向个人开发者。安全审计不做企业治理面板，开发工具不做团队协作/代码评审集成。

**决策二：预警雷达定位。** 安全审计 Skill 仍然是"安装前的风险雷达"——L1/L2 不做深度红队测试，L3 的 LLM 意图分析是最深的能力。版本感知仍然是"变更感知"——不锁定版本，只告知变更。

**决策三：零侵入适配。** 行为测试运行器的 evals.json 是可选的——Skill 作者不写 evals.json 也能用 L1 的质量检查。安全审计 Skill 对所有 Skill 零侵入——纯文档 Skill 也能被扫描。

### 8.3 与 solutions v2 的关系

本方案不是替代 solutions v2，而是在其基础上扩展。solutions v2 的四层架构（规范层、依赖感知层、安全预警层、版本感知层）仍然有效——它们是能力维度的分解。本方案增加的是用户维度的分解（消费者侧 vs 生产者侧），以及新的能力模块（脚手架、行为测试运行器、质量评分）。

solutions v2 的分层产品形态（L1/L2/L3）、与 x-cmd 的互补共生策略、零侵入适配原则、资源整合为主的实施策略——这些核心设计全部保留。

## 九、总结

Agent Skill 基础设施化的完整方案围绕两条产品线展开：安全审计 Skill（消费者侧）和开发工具 Skill（生产者侧）。两条线共享 L1/L2/L3 分层设计，每层独立可用、渐进增强。

最有价值的切入点是**行为测试运行器**——这是当前生态中唯一的完全空白点（成熟度 2/10），覆盖 feasibility 报告中"测试框架空白"（严重度 7/10）的核心痛点。实现方式是在 L2 层通过脚本调用 Agent 运行时（Claude Code / OpenCode CLI），自动执行 evals.json 中定义的测试用例，判定触发准确性、流程正确性和输出规范性。

安全审计 Skill 的 L1 可以通过 x-cmd/skill 仓库分发，建立社区生态位；L2/L3 保持独立分发。开发工具 Skill 的 L1 同样可以通过 x-cmd 分发；L2 的行为测试运行器是最强差异化能力。

实施路径：第一阶段（0-3个月）交付两个 L1 Skill（x-cmd 分发）+ L2 行为测试运行器原型，第二阶段（3-6个月）完善 L2 安全审计和开发工具，第三阶段（6-12个月）交付 L3 智能增强 + 生态扩展。
